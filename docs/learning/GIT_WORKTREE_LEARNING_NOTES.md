# Git Worktree 学习笔记（结合本项目实践）

更新时间：2026-03-25  
适用对象：非科班开发背景、需要在真实项目中理解 Git 协作方式的业务/分析同学

## 1. 一句话理解

Git 不只是“保存代码历史”的工具，它更像是一个帮助我们管理“多条开发路线”和“多处施工现场”的系统。

在这个体系里：

- `branch` 是开发路线
- `worktree` 是施工现场
- `commit` 是阶段性存档
- `merge` 是并回主线
- `thread` 是讨论空间，不是代码空间


## 2. 用电脑类比理解 Git 核心概念

### 2.1 Repository（仓库）

- 类比：一个总项目文件夹 + 历史档案系统
- 里面不仅有文件，还有完整的版本历史

### 2.2 Branch（分支）

- 类比：从主线分出去的一条单独开发路线
- 比如：
  - `main`：正式主线
  - `feature/dashboard-redesign`：仪表盘改版路线
  - `feature/db-migration`：数据库迁移路线

它是“逻辑路线”，不一定天然等于一个新的物理文件夹。

### 2.3 Worktree（工作树）

- 类比：给某一条 branch 单独开出来的一个施工现场 / 办公区 / 实体文件夹
- 它是真实存在的工作目录

作用：

1. 同一个仓库可以同时开多个工作现场
2. 不同任务可以并行施工
3. 避免一个目录里来回切分支导致状态混乱

### 2.4 Commit（提交）

- 类比：阶段性存档点
- 记录某一时刻你这条开发路线的改动内容

### 2.5 Merge（合并）

- 类比：把一条开发路线的成果并回主线


## 3. Thread 和 Worktree 的区别

这是最容易混淆的地方。

### 3.1 Thread（线程 / 对话）

- 类比：会议室 / 讨论空间
- 用于沉淀问题、上下文、决策和需求
- 它管理的是“聊天与认知”

### 3.2 Worktree（工作树）

- 类比：施工现场 / 实体工作目录
- 用于承载真实代码改动
- 它管理的是“文件与版本”

### 3.3 一句话区分

- `thread` 解决的是“我们在聊什么”
- `worktree` 解决的是“我们在哪里改代码”


## 4. 用本项目举例理解

假设当前 RM 项目有三个并行方向：

1. 数据库迁移
2. Dashboard 改版
3. 登录安全修复

那么可以这样组织：

```text
main（主线）
│
├─ branch: feature/db-migration
│  └─ worktree A（数据库迁移施工现场）
│
├─ branch: feature/dashboard-redesign
│  └─ worktree B（Dashboard 改版施工现场）
│
└─ branch: hotfix/login-security
   └─ worktree C（登录安全修复施工现场）
```

这时：

- `main` 是稳定主线
- 每个 `branch` 是一条独立路线
- 每个 `worktree` 是对应路线的实体工作区


## 5. 为什么需要 Worktree

如果没有 worktree，常见做法是：

1. 在同一个目录里频繁切分支
2. 一个需求没做完，又切去做另一个需求
3. 很容易出现文件状态混乱、改动交叉污染

而 worktree 的价值就在于：

1. 并行开发多个需求
2. 降低不同需求互相污染的风险
3. 让每个任务的 diff 更清晰
4. 更容易回滚、提交和审查


## 6. Worktree 能解决什么，不能解决什么

### 6.1 能解决的

1. 任务隔离
2. 施工现场隔离
3. 并行推进多个需求
4. 减少在同一目录里频繁切分支带来的混乱

### 6.2 不能解决的

1. 不能自动消灭代码冲突
2. 不能自动保证业务逻辑正确
3. 不能替代设计评审、口径对齐和测试

也就是说：

- worktree 解决的是“开发过程的秩序”
- 不是“自动帮你做正确决策”


## 7. 为什么仍然会冲突

即使两个需求分别在不同 worktree 里开发，如果：

- 都修改了同一个后端文件
- 都改了同一个函数
- 都调整了同一段业务规则

那么最后在 merge 回主线时，仍然可能冲突。

Git 能解决的是：

- 文本层面的差异识别

Git 不能自动替你决定的是：

- 哪个业务规则才是正确的
- 哪种实现应该保留


## 8. 正确的工程思路

### 8.1 如果只是探讨

- 开新 `thread` 就够了

### 8.2 如果准备并行开发

- 更适合开新 `branch`
- 如果还要独立施工，再配合 `worktree`

### 8.3 如果既要分讨论，又要分代码

- 最理想：`新 thread + 新 branch + 新 worktree`


## 9. 四个关键词的最短记忆法

1. `thread`
- 我们在哪个会议室讨论

2. `branch`
- 我们走的是哪条开发路线

3. `worktree`
- 我们在哪个施工现场改代码

4. `commit`
- 我们在这条路上存了哪些阶段成果

5. `merge`
- 我们什么时候把成果带回主线


## 10. 回到本项目的实际启发

对 RM 项目来说，未来如果遇到这种场景：

1. 主线在做数据库迁移
2. 同时要试 Dashboard 新设计
3. 还要修一个紧急登录问题

最稳妥的做法不是把所有事情都塞在一个目录里，而是：

1. 主线留在 `main`
2. 每个独立方向开自己的 `branch`
3. 有并行施工需求时开对应 `worktree`
4. 最后通过 `commit + merge` 归回主线


## 11. 一句话总结

`branch` 决定你走哪条路，  
`worktree` 决定你在哪个现场施工，  
`thread` 决定你在哪个会议室讨论。
