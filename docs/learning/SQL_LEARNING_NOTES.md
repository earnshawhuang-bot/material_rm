# SQL 学习笔记（结合本项目实践）

更新时间：2026-03-24  
适用对象：从 Excel / SharePoint / PBI 走向数据库化协作的业务与分析同学

## 1. 先用一句话理解 SQL 数据库

SQL Server 更像是一个“数据操作系统”，不是单纯存文件的网盘。  
它不仅存数据，还负责权限、约束、查询性能、并发、审计与可追溯。


## 2. 用“电脑文件系统”类比数据库架构

1. 实例（Instance）
- 类比：一台电脑
- 你当前连接的是服务器 `172.18.164.9`

2. 数据库（Database）
- 类比：电脑里的一个独立项目文件夹（但能力更强）
- 例如：`master`、`PPICDW`、未来的 `rm_inventory_db`

3. Schema
- 类比：项目文件夹下的子目录（如 `dbo`、`ODS`、`EDW`）
- 用于对象分域、权限隔离、命名规范

4. 表（Table）
- 类比：结构化 Excel 表
- 真正承载数据（行、列）

5. 视图（View）
- 类比：保存好的筛选结果页 / 预定义报表数据集
- 常用于给 BI 稳定读取口径

6. 存储过程（Stored Procedure）
- 类比：数据库内部的自动化脚本
- 适合 ETL、汇总、批处理任务

7. 函数（Function）
- 类比：可复用公式
- 用于复用业务计算规则

8. 索引（Index）
- 类比：书本目录
- 不改变数据本身，但显著提升查询速度

9. 约束（Constraint）
- 类比：数据录入规则
- 防止脏数据（如唯一键、非空）


## 3. 系统库与业务库怎么理解

系统库（安装 SQL Server 实例时自动创建）：

- `master`：全局元信息与登录映射
- `model`：新建数据库模板
- `msdb`：任务调度与作业历史
- `tempdb`：临时计算工作区（重启后重建）

业务库（人工创建）：

- 比如 `PPICDW`、未来的 `rm_inventory_db`

关系：

- 业务数据逻辑上独立存放在业务库里
- 运行层共享同一实例资源（权限、作业、临时计算等）


## 4. SharePoint + PBI + SQL + Python 的分工建议

你当前典型流程：

- 同事从 SAP 导出 Excel -> 上传 SharePoint
- PBI 或 Python 做 ETL
- 再建模、DAX、可视化

推荐职责拆分：

1. SharePoint
- 擅长：文件协作入口
- 短板：强治理、审计、复杂口径沉淀较弱

2. Python
- 擅长：复杂清洗、文本处理、接口调用、自动化编排
- 短板：团队成员可读性与统一治理依赖工程规范

3. SQL（SQL Server）
- 擅长：标准化落库、统一口径、并发查询、权限审计、历史追溯
- 短板：初期建模与治理成本高于 Excel

4. PBI
- 擅长：语义建模、DAX、可视化、业务消费
- 短板：不应承担过多重 ETL（维护成本会快速上升）

结论：

- 不是互相替代，而是分层协同
- 推荐“SharePoint 入口 + Python/ETL 清洗 + SQL 沉淀 + PBI 展示”


## 5. “PBI 能做的 ETL，SQL 都能做吗？”

结论：大部分可以，但不等于全部一模一样。

- SQL 更强于关系型集合计算、稳定生产化口径
- Python 更强于非结构化和复杂自定义处理
- PBI 更适合消费而非承载重治理 ETL


## 6. 结合本项目的落地启示

目标：建设独立业务库 `rm_inventory_db`（与 `PPICDW` 同级）

建议最小对象：

1. 6 张核心表（库存快照、action、mapping、用户、上传日志、枚举）
2. 至少 1 个面向 BI 的视图
3. 清晰主键/唯一键与索引策略

关键业务规则（必须固化）：

1. SAP 上传按 `snapshot_month + rm_type` 覆盖
2. Mapping 整表替换后回填历史
3. Action 跨月继承 + 导入“补空不覆盖”

3. SQL：沉淀可信数据与统一口径  
4. PBI：建模与呈现管理层视图
